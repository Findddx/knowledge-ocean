# 21 · 经典案例研究与决策记录

## 定位

经典案例研究的价值，不是讲故事，而是把 `症状 -> 分层 -> 证据 -> 判断 -> 方案 -> 回滚` 这条链条固定下来。真正成熟的硬件知识库，应该能让后来的人看见你为什么这么判断，而不是只看见最终配置。

## 学习目标

- 能把一次扩容、故障或采购选择写成可复盘的证据链，而不是经验帖。
- 能区分症状、事实、推断、决策和后续观察项。
- 能为存储硬件案例同时保留 OS、控制器、BMC、采购和业务影响证据。
- 能用 ADR/决策记录沉淀“为什么选 A、不选 B、如何回滚”。

## 核心直觉

案例页的价值在于复用判断过程。后人不只需要知道你买了哪块盘、配了哪种 RAID，更需要知道当时有哪些约束、证据来自哪一层、哪些未知被接受、什么条件触发回滚。

## 先抓住六个问题

1. 这个案例的核心冲突是 `兼容性`、`性能`、`可靠性` 还是 `采购`？
2. 证据来自哪一层：物理、控制器、固件、OS、应用，还是市场？
3. 最终方案优化的是 `短期可用`，还是 `长期可维护`？
4. 有没有回滚路径？
5. 有没有把“不知道”的部分明确记录下来？
6. 这个结论能不能迁移到下一台机器、下一次采购或下一次排障？

## 机制拆解

| 阶段 | 写什么 | 不要写成 |
| --- | --- | --- |
| 症状 | 用户或系统最先看到的问题 | 直接下结论 |
| 分层 | 物理、背板、控制器、OS、应用、采购分别有什么证据 | 把所有问题都叫磁盘问题 |
| 证据 | 命令输出、BMC 截图摘要、控制器状态、厂商文档链接 | 只写“看起来正常” |
| 判断 | 哪些事实支持候选方案，哪些未知仍存在 | 事后合理化 |
| 决策 | 选 A 的理由，以及不选 B/C 的理由 | 只贴最终配置 |
| 回滚 | 失败条件、恢复步骤、观察窗口 | 没有退出机制 |

## 案例研究不是总结，而是复盘结构

### 一个合格案例至少要有六部分

1. 背景与目标
2. 已知约束
3. 证据清单
4. 候选方案
5. 决策与原因
6. 后续观察点

## 经典案例 1：通用 2U 服务器存储扩容为什么不能只看空盘位

### 表面症状

- 机器前面还有空位，看起来“再插几块盘就行”。

### 真正问题

- 一台 2U 服务器是否具备对应背板、线缆、控制器和前后盘位配置，不能靠肉眼猜。
- 以 Dell PowerEdge R750 规格页为例，同一代 2U 机型也可能存在 12 x 3.5、8 / 16 / 24 x 2.5，以及前后盘位与 NVMe/SAS/SATA 的多种不同组合。

### 应当收集的证据

- Service Tag
- 当前 backplane 配置
- PERC / HBA 型号
- iDRAC inventory
- Linux `lsblk` / `lspci`

### 决策原则

- 先确认“平台是否支持”，再谈“买什么盘”。

## 经典案例 2：RAID 卡后的 Linux 设备名为什么会误导人

### 表面症状

- Linux 里只看到一个大块设备，于是误以为只有一块盘。

### 真正问题

- OS 看到的可能是 PERC 暴露出的 `Virtual Disk`。
- 物理盘数量、状态、重建过程要回到控制器或带外管理里看。

### 应当收集的证据

- PERC Physical Disk / Virtual Disk 清单
- Linux 设备节点
- 控制器日志
- Redfish / iDRAC 存储视图

### 决策原则

- 以后所有“盘数量”“坏盘”“扩容”类问题，都必须同时保留控制器视角与 OS 视角。

## 经典案例 3：高容量 QLC SSD 与 HDD 的采购比较不能只看单盘价

### 表面症状

- HDD 单 TB 更便宜，于是看起来总是最省钱。

### 真正问题

- 高容量 QLC SSD 的收益很多时候来自：
- 更高机架密度
- 更低功耗
- 更少盘位占用
- 更简单的数据路径
- 但它的适用前提是：负载足够读密集，耐久模型匹配，平台和预算也允许。

### 应当收集的证据

- 目标数据集增长速度
- 实际写入比例
- 单机盘位数
- 功耗与机架预算
- 备件与保修方案

### 决策原则

- 采购时必须把 `性能 / 容量 / 密度 / 功耗 / 运维复杂度 / TCO` 一起算。

## 经典案例 4：AI 节点里的“存储问题”经常其实是平台问题

### 表面症状

- 训练作业吞吐不稳，以为要换更快 SSD。

### 真正问题

- 也可能是：
- GPU、NIC、NVMe 不在合理本地性上
- PCIe lane 被共享
- 热设计导致降频
- 电源和散热没有留余量

### 应当收集的证据

- GPU / NIC / NVMe 拓扑
- PCIe 本地性
- 功耗与温度遥测
- blk-mq / 网络队列 / 应用吞吐曲线

### 决策原则

- 不要把所有瓶颈都归因到“盘不够快”。

## 一个建议直接复用的案例模板

```text
案例名称：
日期：
目标：

1. 背景
2. 现状与约束
3. 证据清单
4. 候选方案 A / B / C
5. 最终决策
6. 为什么不是其他方案
7. 风险与回滚
8. 后续观察项
9. 关联页面
```

## 决策记录页应该怎么写

### 写“已知”，也写“未知”

- 不能只写最后结论。
- 未知项本身就是以后复盘时最有价值的内容。

### 写“为什么不选 B”

- 如果没有候选方案对比，后人无法判断你的结论是不是偶然。

### 写“回滚条件”

- 例如：
- 一周内若出现温度异常
- 若控制器不识别新盘
- 若重建窗口超出可接受范围
- 若应用延迟明显恶化

## 设计 / 采购判断

案例研究中的采购结论必须明确责任边界：如果选择 OEM 认证件，要记录它换来了哪些支持和遥测能力；如果选择第三方企业盘，要记录验证范围和自担风险；如果选择消费盘，要写清楚为什么这个场景可以接受更弱的 PLP、耐久和保修模型。

## 故障模式

- 证据跳层：看到应用慢就直接归因到 SSD，忽略 PCIe 拓扑、散热、文件系统和网络。
- 案例不可复用：只写结论，没有硬件配置、固件版本、盘位和命令输出摘要。
- 没有反事实：不写为什么没选其他方案，后续无法判断决策质量。
- 未知被隐藏：把没有验证过的兼容性写成确定事实。
- 回滚缺失：新盘、新 RAID 或新文件系统上线后没有退出条件。

## Linux / 硬件观察命令

### 观察 1：把一次真实扩容改写成案例页

- 写背景
- 写证据
- 写候选方案
- 写最终决策

目标：把“做过”变成“可复用”。

### 观察 2：做一张分层证据表

- L0-L11 每层填一条当前案例证据

目标：训练自己不要跳层下结论。

### 观察 3：为一台待扩容服务器建第一份 ADR

- 主题可以是：
- 存储扩容
- 是否用 RAID
- 是否买 OEM 认证盘
- 是否先上 HDD 还是先上 SSD

目标：让知识库开始积累真正的决策历史。

### 观察 4：为案例采集最小命令证据

```bash
date -Is
hostnamectl
lsblk -o NAME,SIZE,TYPE,ROTA,MODEL,SERIAL,FSTYPE,MOUNTPOINT
lspci -nn | grep -Ei 'raid|sas|sata|storage|nvme'
sudo nvme list 2>/dev/null || true
sudo smartctl -a /dev/sdX 2>/dev/null || true
journalctl -k --since "24 hours ago" | grep -Ei 'nvme|scsi|sas|raid|blk|xfs|ext4|error|reset'
```

目标：让案例至少能复现当时的硬件对象、内核证据和时间线。

## 前沿趋势

- 决策记录会越来越依赖机器可采集证据：Redfish inventory、NVMe log、SMART、PERC/RAID 事件和监控时间线应能互相交叉验证。
- AI 节点和高密度 NVMe 节点的案例不能只写“换盘”，还要纳入 PCIe lane、NUMA、本地性、风道、功耗和网络拓扑。
- 存储采购 ADR 应逐步变成平台生命周期记录，覆盖固件批次、兼容矩阵、扩容节奏和替换策略。
