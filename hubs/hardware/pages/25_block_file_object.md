# 25 · 块、文件、对象存储与数据服务语义

## 定位

很多人把存储系统理解成“盘快不快、容量够不够”，但应用真正面对的首先不是盘，而是 `数据服务语义`。同样一份数据，被呈现为 `块`、`文件` 还是 `对象`，决定了谁管理元数据、谁负责一致性、谁负责共享和扩展。

## 学习目标

- 能从应用视角区分 block、file、object 三种数据服务契约。
- 能解释元数据、一致性、共享、权限、快照和生命周期策略分别在哪一层实现。
- 能判断 SAN、NAS、对象网关和 Ceph RBD/CephFS/RGW 的边界。
- 能避免把对象存储误当作“大号 NAS”，或把块存储误当作天然更快。

## 核心直觉

块、文件、对象不是硬件等级，而是应用拿到的数据接口。块把原始可寻址空间交给主机，文件把目录和共享语义交给文件服务，对象把 bucket/key/metadata 和 API 语义交给对象服务。

## 先抓住六个判断问题

1. 应用拿到的是 `LBA / volume`、`路径 / 文件名`，还是 `object ID / bucket`？
2. 元数据主要由 `主机文件系统`、`文件服务`，还是 `对象服务` 维护？
3. 数据更新是按 `block`、`file range`，还是通常按 `whole object` 处理？
4. 共享是靠主机自己协调，还是由文件服务 / 对象服务提供多客户端语义？
5. 目标场景更看重 `低延迟原始 I/O`、`POSIX/共享目录`，还是 `海量非结构化数据和元数据检索`？
6. 你需要的是一种“存储介质”，还是一种“对应用暴露的数据接口”？

## 机制拆解

| 类型 | 上层看到 | 谁负责元数据 | 典型协议/接口 | 典型场景 |
| --- | --- | --- | --- | --- |
| Block | volume、LUN、块设备 | 主机 OS/文件系统 | NVMe、iSCSI、FC、RBD | VM、数据库、裸盘卷 |
| File | 目录、路径、文件名 | 文件服务端 | NFS、SMB、CephFS | 共享目录、用户文件、训练数据集 |
| Object | bucket、key、metadata | 对象存储服务 | S3 API、Swift、RGW、CDMI | 归档、备份、数据湖、静态资产 |

## 块、文件、对象不是“谁更高级”，而是三种不同契约

### Block Storage

- SNIA 对 block storage 的定义很直接：它是一种把数据存成 blocks 的方式。
- 主机通常拿到的是一个 volume / LUN / block device，然后由主机文件系统决定如何把这些块组织成目录、inode 和文件。
- 所以 block storage 更像“把原始可寻址块空间交给主机”。

### File Storage

- SNIA 对 file storage 的定义同样直接：它是一种把数据存成 files 的方式。
- 文件服务对上层提供路径、目录、权限、文件名和共享语义，典型协议是 `NFS`、`SMB/CIFS`。
- 对应用来说，文件存储的价值常常是共享目录和 POSIX 风格访问，而不是最底层块布局。

### Object Storage

- SNIA 对 object storage 的说明强调：对象是以唯一 object ID 管理的原子单位，通常带有丰富元数据。
- 对象存储更强调 `ID + data + metadata`，而不是目录树和块地址。
- 它通常通过 RESTful API 访问，常见语义是 create / read / delete，很多实现会提供 S3 兼容接口，SNIA 也长期推动 `CDMI` 这类标准管理接口。

## 真正重要的不是术语，而是“谁负责哪一层”

### 在块存储里

- 主机自己负责文件系统、卷扩容、目录语义和很多一致性问题。
- 所以 block storage 更像给数据库、虚拟化、文件系统和本地卷管理提供基础原料。

### 在文件存储里

- 文件服务本身承担目录、锁、共享和权限语义。
- 所以它适合多客户端共享工作目录、用户主目录、内容协作和常规 NAS 场景。

### 在对象存储里

- 对象服务承担对象命名、元数据管理、策略和 API 暴露。
- 应用通常直接面对对象接口，而不是 POSIX 目录树。
- 所以它天然更适合海量非结构化数据、日志归档、备份、媒体、AI 数据集和云原生应用。

## 同一个底层系统，可能同时提供三种接口

### Ceph 是很好的例子

- Ceph 官方架构文档明确说明，它在一个统一系统里同时提供 `object`、`block` 和 `file` 三类服务。
- 更关键的是，这三类服务最终都映射到底层对象存储集群。

### 这给你的启发是

- 块、文件、对象不是互斥的“硬件分类”。
- 它们更像是底层存储系统向上暴露的不同数据服务契约。

### 数据保护责任也会随接口移动

| 接口 | 快照通常在哪里做 | 备份最容易漏掉什么 | 恢复时先验证什么 |
| --- | --- | --- | --- |
| Block | 卷、LUN、RBD image、阵列 | 文件系统一致性、数据库日志、挂载顺序 | 文件系统与应用能否启动 |
| File | 共享文件系统、目录树 | ACL、锁语义、隐藏元数据、目录规模 | 权限、目录完整性和多客户端语义 |
| Object | bucket、object version、存储策略 | 版本、生命周期、metadata、对象锁 | key 列表、对象元数据、访问策略和校验 |

所以第 28-35 章讨论备份和灾备时，不能只问“底层有没有副本”，还要问应用实际面对的是哪种数据契约。

## NAS、SAN、对象网关不要混

### SAN

- SAN 更接近“通过网络提供 block access”。
- 主机拿到的是 volume / LUN / block device，后续怎么组织数据主要还是主机自己决定。

### NAS

- NAS 更接近“通过网络提供 file service”。
- 客户端拿到的是目录和文件语义，而不是底层块地址。

### Object Gateway

- 对象网关更接近“通过 HTTP / REST 暴露对象接口”。
- 客户端拿到的是 bucket、object key、metadata 和 API，而不是路径或 LBA。

## 什么时候选哪一种

### 更适合 Block 的场景

- 数据库
- 虚拟化镜像
- 需要自己控制文件系统和卷管理的应用
- 对低延迟和原始 I/O 路径非常敏感的场景

### 更适合 File 的场景

- 团队共享目录
- 多主机访问同一文件树
- 需要 POSIX / 路径 / 权限语义的通用文件工作负载

### 更适合 Object 的场景

- 海量非结构化数据
- 归档、备份、媒体、日志湖
- 云原生应用
- AI 数据集、模型文件、长生命周期数据

## 服务器落地时最该问的十个问题

1. 应用到底想要 block、file 还是 object 语义？
2. 共享语义由谁负责，主机、文件服务还是对象服务？
3. 元数据管理会不会成为瓶颈？
4. 是需要低延迟块访问，还是更需要多客户端共享？
5. 是否需要 S3 / HTTP 接口，而不是 POSIX？
6. 扩容时是加盘、加 NAS 头、还是加对象节点？
7. 当前监控和备份体系更适配哪一种数据服务？
8. 权限、配额、快照和生命周期策略在哪一层实现？
9. 应用能否容忍 eventual / API-based 访问模型？
10. 你在采购的是“存储介质”，还是“上层数据服务能力”？

## 设计 / 采购判断

- 数据库和虚拟化通常先看 block，因为它们希望自己控制文件系统、缓存、一致性和低延迟 I/O。
- 多客户端共享目录、用户 home、工程文件和通用 NAS 场景优先看 file，因为服务端承担共享和权限语义。
- 归档、备份、日志湖、媒体、AI 数据集和云原生应用优先评估 object，因为元数据、生命周期和横向扩展更重要。
- 同一个底层系统可以同时提供三类接口，但容量池、QoS、恢复流量和元数据服务会互相影响，不能只按接口名采购。

## 常见误区

### 误区 1：对象存储就是“更大的 NAS”

- 错。对象存储的名字空间、访问协议和更新模型都不同。

### 误区 2：块存储就是“更底层所以一定更快”

- 不一定。它更底层，但最终性能取决于协议、网络、实现和工作负载。

### 误区 3：文件存储只是 block 上面多一层目录

- 过于简化。文件服务还承担共享、锁、权限、元数据和客户端语义。

### 误区 4：底层用对象，就代表应用一定直接面对对象接口

- 错。很多系统会在对象之上再提供 block 或 file 服务。

## 故障模式

- 语义错配：把需要 POSIX rename/lock 的应用迁到对象存储，出现一致性和更新模型问题。
- 元数据瓶颈：文件服务目录树、对象 bucket listing 或小文件工作负载压垮元数据路径。
- 共享责任不清：多个主机同时写同一个块卷，没有集群文件系统或锁协调。
- 备份策略错层：只备份底层块，不理解上层对象版本、生命周期或文件权限。
- 成本误判：对象存储单 GB 便宜，但请求费、生命周期、恢复时间和应用改造成本被忽略。

### 观察 4：在 Linux 上识别当前服务语义

## 前沿趋势

- 对象存储继续成为备份、归档、数据湖和 AI 数据集的默认底层，但应用要适配 API、元数据和生命周期语义。
- Ceph、云存储和企业阵列会越来越常见地在同一底层提供 block/file/object 多协议入口，难点转向 QoS、隔离和恢复流量。
- S3 兼容成为事实生态，但管理、访问控制、生命周期、版本和合规能力仍要逐项验证，不能只看“兼容”两个字。
