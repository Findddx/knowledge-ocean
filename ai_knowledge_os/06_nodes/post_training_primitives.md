# 后训练原语：SFT、DPO、RFT 与 GRPO

> 预训练让模型“懂很多”，后训练决定这些能力以什么行为形式出现。真正决定可用性的，往往不是模型在语料里见过什么，而是系统后来又奖励了什么、压制了什么、要求它以什么方式稳定地表现出来。

## 1. 为什么真正决定“可用行为”的，往往不是预训练而是后训练

一个模型可能知道很多知识，但这并不意味着它会在你需要的时候、以你需要的方式、稳定地把这些能力表现出来。它可能会推理，却不愿意按结构输出；可能知道规则，却在细微偏好上摇摆；可能有领域知识，却在关键判断上不够稳。

从这个角度看，后训练真正解决的，不是“让模型凭空学到一切”，而是把潜在能力压缩成更可靠、更可控、更贴近目标函数的行为模式。InstructGPT 的结果之所以值得反复记住，不是因为某个具体数字，而是因为它证明了一件事：行为塑形本身就是能力交付的一部分。

## 2. 后训练到底在调什么

如果把后训练看成一条连续的问题链，会更容易抓住本质。

### 2.1 什么算好回答

这是 `SFT` 最直接回答的问题。你拿出一组 prompt 和“已知足够好”的输出，让模型学习：面对这类输入，怎样的行为才是你真正想要的。

### 2.2 哪个回答更好

这是 `DPO` 和偏好优化类方法回答的问题。很多任务没有唯一标答，但人类能在两个答案之间表达稳定偏好：哪个更清楚、更贴合语气、更有帮助、更像目标风格。

### 2.3 怎样通过试错逼近高价值行为

这是 `RFT` 与 `GRPO` 等强化式后训练回答的问题。当任务质量可以被 grader 或 reward 函数评价时，模型就不再只是模仿示范，而可以通过采样、打分、更新的循环，把高分行为逐步变得更可能。

这三类问题并不彼此排斥，而是从示范、到偏好、再到策略优化的层层递进。

## 3. 历史线索：从人类反馈到可编程评分

回看对齐研究的发展，会看到一条很清楚的路线。

- `Learning to summarize from human feedback` 证明了可以通过人类比较数据训练奖励模型，再用强化学习优化摘要行为。
- `InstructGPT` 把这种思路带到了更一般的 instruction-following：先收集 demonstrations 做监督微调，再收集排序数据做偏好学习，最后用 RLHF 继续塑形。
- OpenAI 当前的 `RFT` 文档则把这条路线进一步推进为“可编程评分”：你不一定总依赖人工偏好，也可以直接用 grader 把任务质量形式化，再让模型围绕这个评分函数更新。

因此，后训练的发展史，本质上是监督信号越来越接近真实任务目标的发展史。

## 4. SFT：最朴素，也最常被误解

### 4.1 它在机制上到底做了什么

OpenAI 的 SFT 文档把它定义得非常直接：用 example inputs 和 known good outputs 训练模型，让它更可靠地产生你想要的风格与内容。它不是在“神秘对齐”，而是在做示范式教学。

### 4.2 SFT 真正擅长什么

SFT 特别适合处理这些问题：

- 输出格式不稳定；
- instruction-following 经常偏航；
- 想把模型迁移到某种明确风格或结构；
- 希望小模型复现大模型某类稳定输出模式。

OpenAI 的 SFT 文档也把典型适用场景写得很清楚：分类、细腻翻译、特定格式生成、修正 instruction-following failures。

### 4.3 SFT 真正改变的不是知识量，而是默认姿势

很多人一提 SFT 就想到“喂更多领域数据”。这不完全错，但更关键的是：SFT 会改变模型面对某类任务时的默认行为姿势。它让模型更倾向于以某种结构、语气、局部流程来回答，而不必每次都靠 prompt 反复提醒。

### 4.4 它的边界在哪里

SFT 的限制也很清楚：

- 它学到的是示范分布，不是抽象奖励本身；
- 数据覆盖不足时，模型很容易学会表面模式，却学不会更深层判断；
- 它擅长“把姿势立住”，但不天然擅长复杂长程策略。

SFT 文档本身就把流程写成四步：先构建数据集来定义什么叫“good”，最后再用 fine-tuned model 评估结果。这个顺序很重要，因为它提醒你：SFT 成败首先取决于你是否真的定义清楚了什么算好行为。

## 5. DPO：偏好优化不是教标答，而是教排序结构

### 5.1 它为什么重要

现实中有很多任务并没有唯一标准答案，却明显存在“更合适”的答案。比如摘要该突出什么、客服回复该多有温度、文案该偏专业还是偏亲切。对这类任务来说，比较关系往往比标答更接近真实需求。

### 5.2 它在机制上做了什么

`DPO` 论文的关键洞见，是把原本 RLHF 里“奖励模型 + 强化学习”的问题，改写成可以直接在偏好对上优化排序倾向的形式。OpenAI 的 DPO 文档则把这件事落到了非常实用的层面：每个样本由 prompt、preferred output、non-preferred output 组成，模型要学的不是“唯一正确”，而是在同一任务下把更被偏好的回答概率拉高。

### 5.3 DPO 擅长什么

DPO 特别适合这些情形：

- 风格、语气、取舍有明显偏好；
- 人类能稳定比较两个回答的好坏；
- 你知道“哪个更好”，但不想把这种判断写成复杂的 grader。

### 5.4 DPO 的边界在哪里

它不天然擅长长程信用分配，也不天然解决多步策略问题。如果结果质量依赖工具调用、复杂推理或长期回报，仅靠偏好对往往不够。它更像一把精修刀，而不是重型引擎。

### 5.5 为什么 OpenAI 建议把 SFT 和 DPO 叠起来

OpenAI 当前的 DPO 文档明确建议：先用 preferred responses 的一个子集做 SFT，再用这个 SFT 模型做起点继续 DPO。这条建议非常值得记住，因为它说明真实工程里常见的关系不是“SFT 或 DPO 二选一”，而是先立住正确姿势，再用偏好关系做细修。

## 6. RFT：当目标可以被 grader 描述时，训练开始从模仿走向策略

### 6.1 它的本质

OpenAI 的 RFT 文档把它定义为：使用你定义的反馈信号去适配 reasoning model。和 SFT 不同，RFT 不再依赖固定正确答案，而是对每个 prompt 采样多个候选回答，通过 grader 打分，再基于这些分数做 policy-gradient 更新。

这意味着优化目标发生了范式转换：模型不再主要学“像不像示范”，而是学“怎样更可能得到高分”。

### 6.2 为什么 grader 才是 RFT 的真正核心

RFT 的难点从来不只是“把训练跑起来”，而是你能不能把任务质量写成一个足够可信的评分函数。OpenAI 文档在 “When should I use reinforcement fine-tuning?” 一节说得很直白：

- RFT 最适合 `unambiguous tasks`；
- 合格专家应当能对答案形成较高一致性；
- 任务最好是 `guess-proof`，否则奖励信号会太噪。

这些条件连在一起，其实在说：RFT 的本体不是模型，而是 grader 的质量。

### 6.3 它最适合什么任务

OpenAI 文档给出的例子非常典型：

- 复杂且高要求的专业推断；
- 医疗、法律等需要细规则判断的场景；
- 数学、代码、抽取、分类等可验证任务；
- 需要复杂 reasoning，但终局质量能被相对稳定地评分的任务。

### 6.4 它最大的风险是什么

如果 grader 不够稳，模型很可能学会对 grader 投机，而不是学会真正更好的行为。它可能：

- 学会拉长答案冒充更认真；
- 学会迎合局部评分规则却损害整体质量；
- 学会通过猜测拿到偶然高分。

所以 RFT 最大的风险不是“训练太贵”，而是“你把噪声写成了制度”。

## 7. GRPO：为什么它在 reasoning 任务里特别引人注意

### 7.1 它从哪里来

`DeepSeekMath` 把 `GRPO` 介绍为 `PPO` 的一个变体，核心做法是通过 group-relative 的方式估计优势，从而在 reasoning 任务上提升训练效率与稳定性，并缓解一部分内存开销问题。

### 7.2 它真正告诉了我们什么

GRPO 的价值不在于又多了一个缩写，而在于它进一步说明：当任务进入可验证推理领域，后训练就越来越像策略搜索。此时采样质量、相对奖励、内存效率、训练稳定性都不再是配角。

### 7.3 什么时候该把 GRPO 当作重点考虑

当任务具备这些特征时，GRPO 会显得更自然：

- 输出可验证；
- 多个候选之间能形成有意义的相对比较；
- reasoning 过程比单轮模仿更重要；
- 训练效率与稳定性已经变成首要工程问题。

如果这些前提不成立，GRPO 的吸引力就会快速下降。

## 8. 四种原语不是替代关系，而是层次关系

把 SFT、DPO、RFT、GRPO 视为互斥选项，是初学者最常见的误区。更贴近真实工程的理解是：

1. `SFT` 先把基础行为、格式和风格立住；
2. `DPO` 再把细微偏好与主观取舍校准到更想要的方向；
3. `RFT / GRPO` 在可评分、可验证、需要复杂策略的任务上继续提高。

如果一定要用一句话概括这层关系：

- SFT 解决“姿势”；
- DPO 解决“口味”；
- RFT / GRPO 解决“策略”。

## 9. 一个更可迁移的选型框架

| 你遇到的问题 | 更可能合适的原语 | 为什么 | 先决条件 |
| --- | --- | --- | --- |
| 输出格式总不稳，常不按要求回答 | SFT | 需要先把默认姿势立住 | 有高质量示范 |
| 两个回答都不算错，但你明显偏好其中一种 | DPO | 你需要优化的是排序关系 | 有稳定偏好对 |
| 任务质量可被 grader 评价，且 reasoning 很关键 | RFT | 你优化的是得分，不只是模仿 | 任务不含糊，grader 可信 |
| 可验证推理任务，需要更高 RL 效率 | GRPO | 你已经进入策略搜索问题 | 结果可验证、采样可比较 |
| 失败主要因为检索、上下文、工具设计差 | 先别训练 | 这不是后训练原语的首要问题 | 先修 workflow 与 eval |

这张表最重要的用途，不是替你拍板，而是强迫你先把“我到底在优化什么”说清楚。

## 10. 一个案例：同一个合规审阅模型，为什么会分三步走

想象你在做一个合规审阅助手。

### 第一步：模型总是不按结构输出

这时最先需要的往往不是偏好优化，而是 SFT。因为你面临的是“默认姿势不对”，不是“偏好还没校准”。

### 第二步：结构基本对了，但语气、取舍和解释风格还不理想

这时 DPO 往往更合适。因为你要调的不是“会不会输出”，而是“哪种输出更符合你的偏好”。

### 第三步：系统需要根据大量规则做高要求判断

如果此时你已经能构造相对稳定的 grader，RFT 才开始真正值得考虑。因为这时优化目标已经从表层行为变成“最后的判定质量”。

这个案例真正想说明的是：后训练不是一次性挑一招，而是随着问题类型升级，训练目标也在升级。

## 11. 常见误判：为什么很多团队会过早或错误地使用后训练

### 11.1 把知识缺失错当成偏好问题

如果模型连关键事实都拿不到，DPO 不会神奇地把缺失知识补出来。

### 11.2 把工作流问题错当成训练问题

如果失败主要来自上下文拿不到、工具 schema 模糊、检索质量差，过早 fine-tune 往往只是在错误工作流上做补丁。

### 11.3 在任务仍然含糊时就上 RFT

RFT 最怕的是任务标准本身不稳。专家自己都说不清什么算好，模型只会学会放大噪声。

### 11.4 因为 GRPO 热门就硬套到一切任务上

GRPO 最有价值的前提，是任务具备可验证推理结构。没有这个前提，它的收益会迅速消失。

## 12. 一个诊断表：看到症状时先怀疑什么

| 症状 | 先怀疑 | 解释 |
| --- | --- | --- |
| 结构经常错、格式不稳 | SFT 数据或任务定义不足 | 默认姿势没立住 |
| 回答都差不多对，但明显“味道不对” | DPO 更合适 | 你需要的是偏好校准 |
| 答案长度越来越长，却不一定更对 | RFT grader 可能被投机 | 模型在学会骗分 |
| 多步 reasoning 提升有限 | 仅靠 SFT / DPO 不够 | 你可能已经进入策略问题 |
| 换了训练方法也没明显进步 | 问题不在后训练 | 先检查 retrieval、context、tools、eval |

## 13. 练习题

1. 一个模型能基本按模板输出，但总在“解释得是否足够稳健”上摇摆。你会优先考虑 SFT、DPO 还是 RFT？为什么？
2. 一个任务专家之间经常意见不一致，但团队仍想做 RFT。这里最大的风险是什么？
3. 你如何判断一个失败是“模型不会”，还是“workflow 没设计好”？

## 14. 一个更深的理解：后训练其实是在做制度设计

如果从更抽象的角度看，预训练像让模型接触世界、吸收统计规律；后训练更像给模型建立制度。制度的作用，不是创造智力本身，而是规定：

- 什么行为会被鼓励；
- 什么行为会被压制；
- 什么样的输出会在长期迭代中变得更可能。

SFT 像导师示范，DPO 像同题比较，RFT 像带评分标准的反复练习，GRPO 则像在同一组解法中学习哪种推理更值得保留。这样理解以后，你就不会再把后训练看成一组彼此孤立的算法，而会把它看成一套塑形制度。

## 15. 结语：先定义你想奖励什么，再决定怎么训练

后训练最值得长期记住的一条原则，不是哪种算法在某个榜单上赢了，而是：你最终得到的，往往不是“最聪明的模型”，而是“最认真优化了某种目标的模型”。

因此，真正重要的问题永远排在算法之前：

1. 我要塑造的是格式、偏好，还是策略？
2. 我的监督信号是否足够可信？
3. 我的 eval 能否分辨改进与退化？
4. 这个问题到底该靠训练解决，还是该先修 workflow？

能把这四个问题想清楚，后训练才会从“会几个缩写”变成真正的研究与工程能力。
