# Context Engineering、长上下文与记忆设计

## 1. 真正的问题，往往不是模型太弱，而是“世界”喂得太乱

当一个 agent 或知识工作流表现不佳时，人们最习惯的解释是：模型不够强、上下文窗口不够大、推理能力还不够。这个解释有时成立，但在真实系统中，更常见的失败来自另一个地方：模型收到的并不是一个可推理的世界，而是一堆未经整理的信息碎片。

模型看到的“世界”，其实就是你送进上下文窗口的内容及其顺序、类型、边界和更新方式。换句话说，上下文并不是一个被动容器，而是模型当前可感知现实的全部。现实被喂得混乱，推理就会在混乱上继续展开；现实被喂得有结构，模型才有机会表现出你期待的能力。

这就是为什么到了 2026 年，仅仅谈 `prompt engineering` 已经不够。你要设计的不再只是一段提示词，而是一整套信息供给系统：哪些内容必须常驻，哪些内容应该按需检索，哪些内容只该以摘要形式出现，哪些内容必须保留原文，哪些状态应该写回到会话外部。

## 2. 从 prompt engineering 到 context engineering，变化到底发生在哪里

`Prompt engineering` 更接近语言层面的工作。你关注的是指令清不清楚、格式约束够不够明确、few-shot 示例是否贴切、输出模板有没有歧义。它解决的是“如何把要求说清楚”。

`Context engineering` 则更接近系统层面的工作。你关注的是：

- 哪些信息值得进入当前窗口；
- 它们以何种顺序和边界进入；
- 静态信息与动态信息如何分层；
- 什么应该在窗口内保留，什么应该被外置到检索、记忆或文件系统；
- 会话增长时，状态如何被压缩、缓存、续接与回写。

两者的关系不是替代，而是升级。前者处理“句子怎么写”，后者处理“信息怎么流”。如果说 prompt engineering 像写说明书，那么 context engineering 更像设计操作系统的内存和进程管理。

## 3. 一个更严格的定义：上下文是受预算约束的工作记忆

为了避免“上下文”这个词过于宽泛，可以给它一个更精确的定义：

> 上下文是模型在当前一步推理中可直接访问的、受 token 预算约束的工作记忆。

这个定义里有三个关键词。

### 可直接访问

只有已经进入上下文窗口的内容，才是真正被模型“看到”的内容。检索库里有资料、磁盘上有文件、数据库里有结果，并不等于模型已经拥有它们。只要还没有被正确注入，它们对当前推理就是不存在的。

### 工作记忆

上下文最接近人的工作记忆，而不是长期记忆。它适合承载当前任务必须同时考虑的对象，但不适合无限堆积历史。把上下文当作永久仓库，会导致噪音增长、冲突信息累积、成本上升，以及越来越模糊的任务边界。

### 预算约束

上下文始终受制于代价。即使模型提供了很长的窗口，真正昂贵的并不只是“装不装得下”，而是 prefill、缓存命中率、检索质量、冲突事实、位置偏置和延迟。长上下文扩大的是调度空间，不是取消了调度问题。

## 4. 上下文有哪几种成分

把上下文拆成不同类型，是 context engineering 的第一步。最常见的成分至少有四类。

### 稳定前缀：规则、角色与长期约束

这部分包括角色设定、输出格式、安全边界、工具说明、长期工作原则、风格规范等。它的特点是跨请求高度复用，变化慢，适合作为稳定前缀。OpenAI 的 prompt engineering 与 prompt caching 文档都在提醒同一件事：重复使用的内容应尽量稳定并放在前部，这不仅有助于一致性，也直接影响缓存命中和延迟成本。

### 任务状态：当前回合必须接力的中间结构

任务目标、已完成步骤、未完成步骤、用户确认点、中间结论、失败原因，这些内容不属于“知识”，却属于“任务现场”。许多 agent 在多轮后开始走偏，并不是因为它遗忘了知识，而是因为任务状态没有被结构化维护，导致它每一轮都在重新猜自己之前到底做到了哪里。

### 证据上下文：当前任务需要引用的事实材料

检索结果、文档片段、代码段、数据库输出、网页内容、实验结果，都属于证据上下文。它们的特点是与当前子任务强耦合，但不一定需要长期驻留。证据上下文如果组织得好，模型可以做 grounded reasoning；组织得坏，就会变成一堆貌似相关、实则噪声的片段。

### 记忆上下文：跨轮仍有价值的长期事实

用户偏好、项目背景、长期术语约定、已经确认过的事实、可复用的工作协议，这些内容跨轮仍然有意义，但也不应该全部无脑塞进每一轮上下文。它们更适合被维护为外部记忆层，在合适时刻被选择性唤起。

## 5. 长上下文为什么不是“无限记忆”

长上下文的出现，很容易诱发一种危险错觉：既然窗口够长，那我只要把所有材料都丢进去，模型自然会自己找到重点。`Lost in the Middle` 的价值就在于，它系统地提醒我们：模型对长上下文的利用并不均匀。相关信息位于开头或结尾时，模型表现通常更好；当关键证据埋在中间时，性能会明显下降。

这件事的真正启发不是“中间位置有问题”这么简单，而是：上下文不是纯粹容量问题，它有注意力分配问题、位置偏置问题和信息竞争问题。长上下文给了你更大的舞台，但它没有替你编排舞台。

因此，长上下文系统真正需要回答的不是“我能塞多少”，而是：

- 哪些内容必须保留原文；
- 哪些内容只需要引用关键片段；
- 哪些内容应该先做 quote extraction 再进入推理；
- 哪些状态应被压缩成结构化摘要；
- 哪些旧内容已经应当移出窗口，写回到外部系统。

如果不回答这些问题，窗口再长也只是在扩大混乱。

## 6. Anthropic 文档给出的三个建议，为何这么重要

Anthropic 的长上下文文档给出的三条建议，看上去像 prompt 小技巧，实际上触及了 context engineering 的核心。

### 把长文放前面，把问题放后面

这不是排版洁癖，而是在配合模型的读取与聚焦方式。长材料在前，任务请求在后，相当于先建立证据场，再给出求解目标。对多文档分析尤其重要，因为它减少了模型在还没“看完材料”时就过早形成回答框架的概率。

### 用明确标签包住不同文档

用 XML 标签或其他清晰分隔结构标记来源、正文、元数据，本质上是在告诉模型：这些文本并不属于同一层语义空间。没有边界，模型更容易把不同来源混成一个大段落，导致引用混乱、归因错误、证据对应关系模糊。

### 先抽引文，再继续判断

这条建议极其重要，因为它把“推理”分成了两个步骤：先定位证据，再根据证据做解释。对于长文分析、事实性问答、综述撰写、法规比对，这种做法通常比直接让模型“一步到位”更稳健。你可以把它理解为一种轻量级的证据对齐机制。

## 7. RAG 与长上下文不是对立关系，而是不同层级的调度手段

`RAG` 论文最早强调的是：面对知识密集型任务，外部检索可以让生成结果更具体、更事实化。很多人因此把 RAG 当成长上下文的替代品，或者反过来把长上下文当成“无需检索”的更强版本。实际上二者解决的是不同层级的问题。

长上下文解决的是“当前窗口可以容纳更多候选信息”；检索解决的是“在海量候选信息中，哪些值得进入当前窗口”。前者扩大容器，后者决定选材。一个没有检索的长上下文，容易成为冗长杂糅的仓库；一个没有足够窗口和结构的检索系统，则容易把短碎片塞进模型却不给它足够推理空间。

最稳妥的做法，通常是把它们结合起来：先用检索控制进入窗口的材料范围，再在窗口内通过分块、排序、标签、引文提取与结构化状态来提升有效利用率。

## 8. Cache-aware 设计：为什么好的上下文工程也在优化成本

OpenAI 的 prompt caching 文档把一个常被忽略的事实讲得很清楚：上下文设计不只是质量问题，也是成本和延迟问题。缓存命中只发生在精确前缀匹配上；一般需要 prompt 达到 1024 tokens 以上；稳定内容应放在前面，动态内容放在后面；`prompt_cache_key` 可以帮助共享前缀更稳定地路由到相近机器。

这些信息的含义非常深。它告诉你，生产级上下文工程不只是“怎么让回答更聪明”，也是“怎么让系统不反复为相同前缀付钱”。一个稳定的系统前缀、清晰的任务层、变化较晚的用户输入，不仅更利于模型理解，也更利于缓存命中。反过来，如果你每一轮都把相同规则换一种写法、把静态说明夹在动态输入中间，系统会同时损失一致性和经济性。

换句话说，context engineering 在生产环境中至少要优化三个目标：质量、成本、延迟。很多看似“写作习惯”的差异，最终都会出现在账单和响应时间上。

## 9. 记忆设计：不要把上下文当仓库，而要把它当入口

`MemGPT` 把这一点说得很透：有限上下文窗口可以通过类似操作系统分层内存的方式管理，形成“虚拟上下文”。这个比喻之所以有价值，不在于它把系统说得更酷，而在于它提示我们：真正可扩展的 agent，不应该试图把长期记忆全部保留在当前窗口，而应该像操作系统管理快慢存储一样，决定什么应常驻、什么应换入、什么应写回。

从实践角度看，至少可以把记忆分成三层：

| 记忆层 | 典型内容 | 何时进入当前上下文 |
| --- | --- | --- |
| 工作记忆 | 当前任务目标、步骤、局部证据、中间结论 | 几乎每轮都需要 |
| 情景记忆 | 本次项目的重要决策、最近失败原因、已确认约束 | 当前任务延续时唤起 |
| 长期记忆 | 用户偏好、常用模板、长期事实、领域约定 | 被任务显式需要时选择性唤起 |

记忆设计真正困难的地方，在于“选择性”。如果所有长期记忆都自动进入窗口，你得到的是污染而不是帮助；如果长期记忆永远不被唤起，多轮协作又无法形成连续性。记忆层不是越大越好，而是越会选越好。

## 10. 什么时候该保留原文，什么时候该用摘要，什么时候只保留状态

这是上下文工程里最实际也最容易做错的判断之一。

### 应当保留原文的场景

法规、合同、论文关键段落、精确 API 规范、关键实验结果、涉及引用责任的事实材料，通常都更适合保留原文或至少保留可核对片段。因为这些内容一旦被摘要，最危险的损失不是长度，而是语义精度。

### 适合先做引文或摘要的场景

长报告、多篇资料综述、冗长会议记录、网页检索结果集合，往往应先做证据抽取或结构化摘要，再进入综合判断。这里的关键不是“缩短”，而是“让进入窗口的内容已经经过任务导向的压缩”。

### 只适合保留结构化状态的场景

已完成步骤、下一个动作、风险点、用户确认状态、待补资料清单，这些内容如果继续保留在自然语言闲聊中，会造成窗口污染。它们更适合变成简洁、显式、可续接的状态对象。

可以把这个判断记成一句话：事实重精度，材料重筛选，状态重结构。

## 11. 一套可复用的上下文架构

如果要为长期知识工作或 agent 系统设计一个稳定架构，我更推荐这样分层：

### 第一层：稳定前缀

包括角色、目标、输出规范、工具说明、安全边界、长期协议。这一层尽量稳定，方便缓存、方便复用，也方便你知道系统始终在什么框架下工作。

### 第二层：任务状态

包括这次任务的定义、范围、完成条件、已完成步骤、下一步、待确认问题。这一层的目标不是“写得好看”，而是保证下一轮能够无损续接。

### 第三层：按需证据

包括当前轮真正相关的文档片段、检索结果、代码、数据输出。它必须与当前子问题强相关，而不是“顺手把可能有用的都塞进来”。

### 第四层：回写出口

包括知识卡、项目记录、研究笔记、长期记忆、数据库条目、文件变更。没有回写出口的系统，即使当前窗口组织得再好，也仍然在重复劳动。

这套分层的核心思想只有一句话：窗口不是终点，而是中转站。

## 12. 常见失败模式

### 把窗口长度当作能力代理

上下文更长，不等于模型就更能正确组织信息。长度是容量条件，不是充分条件。

### 把检索结果原样拼接

“找到很多资料”不等于“资料已经可用”。没有排序、边界、证据选择和状态绑定的检索结果，只是把噪音合法化。

### 让任务状态淹没在自然语言聊天里

多轮之后开始忘记约束、重复劳动、反复回到已解决问题，常常不是模型变笨，而是状态没有被外显。

### 把长期记忆做成无限追加

长期记忆如果没有淘汰、更新和唤起规则，会逐渐从“帮助系统记住你”变成“让系统带着陈旧偏见工作”。

### 把压缩当作删除

好的压缩是保留结构与关键证据；坏的压缩是把真正决定判断的细节删掉，只留下看上去顺滑的空话。

## 13. 什么时候必须停下来，让人来判断

上下文工程再好，也不能把所有判断外包给模型。以下场景尤其需要人工介入：

1. 任务目标本身仍然模糊，连“什么算成功”都说不清时。
2. 证据之间存在冲突，需要决定相信哪一方时。
3. 摘要或压缩可能带来高风险信息损失时，例如法律、医疗、财务等高风险语境。
4. 系统开始反复在错误结构上继续推理时，此时需要人先修正问题表示，而不是继续追加上下文。
5. 最终输出将触发不可逆行动时，例如对外发布、自动执行、代码落库、权限变更等。

人的作用，不是替模型做所有步骤，而是在关键节点重新定义现实边界。

## 14. 一句总括

长上下文最深的误解，是把它理解成“让模型记住更多”。更准确的理解应该是：它迫使我们学习一种新的能力，去决定什么值得在当下被看见。`Context engineering` 的本质，不是堆字，而是调度；不是灌输，而是编排；不是让模型拥有世界，而是让模型在有限预算里看见一个足够清晰、足够可行动的世界。

## 15. 案例一：一份失败的长文综述 prompt，如何被改造成可推理的上下文系统

最能说明 `context engineering` 价值的，往往不是成功范例，而是失败范例。设想你要比较 6 份关于 `agent eval` 的一手资料，最后产出一份方法备忘录。很多人的第一版 prompt 会长成这样：前面先问“请比较这些资料并给我建议”，然后把 6 份原文或长摘录一股脑贴在后面，再顺手附上前几轮聊天历史。

这种写法常见的失败症状也很稳定：输出非常流畅，但归因混乱；它会抓住开头和结尾的显眼说法，却漏掉中段真正关键的边界条件；它给出的“建议”像是平滑平均，而不是基于证据的裁决。初学者常把这种失败归因于模型不够强，实际上问题更接近“舞台布置错了”。

更稳的改法，不是再多加几个“请仔细思考”，而是把上下文重新编排成四层。

### 第一层：把稳定前缀和任务规则固定下来

先把角色、输出格式、引用责任、禁止猜测的边界写成稳定前缀，并尽量保持跨轮不变。这不仅是为了清晰，也是为了让系统更容易复用与缓存。稳定前缀的职责不是提供“背景知识”，而是规定这场推理应遵守的游戏规则。

### 第二层：把证据材料放到前面，并且明确边界

Anthropic 建议把长材料放前、问题放后，这条建议在综述任务里尤其重要。6 份材料不应被拼成连续大段文本，而应明确包在各自标签中，至少保留来源名、材料类型和时间信息。这个动作看似朴素，却是在告诉模型：这些材料不是一篇文章，而是 6 个需要区分的证据容器。

### 第三层：先抽引文，再进入比较判断

不要第一步就要求“比较并给建议”。先让模型按主题提取每份材料最关键的 2 到 3 条原文证据，记录来源，再开始综合。这相当于先搭证据骨架，再做解释性工作。`Lost in the Middle` 之所以重要，就在于它提醒我们：只靠“全部塞进去”并不能保证关键证据在模型视野里处于高可用状态。

### 第四层：把当前轮的任务状态显式写出来

如果这是一项会跨多轮推进的工作，还应单独维护一个极短状态对象，例如：

```text
本轮目标：比较 6 份资料对 eval 设计的分歧
已完成：证据抽取；初步分组
待解决：哪些分歧来自测量对象不同，哪些来自工程取舍不同
输出要求：一页备忘录，保留引文来源
```

这段状态不是装饰。它是在告诉模型：当前轮不是重新理解全世界，而是接着某个明确进度继续工作。

如果把以上四层组织好，你会发现第二版 prompt 看上去并不一定“更聪明”，但输出会显著更稳。原因不神秘：模型不是突然变强了，而是你终于给了它一个可以推理的世界。上下文工程最深的价值，恰恰在这里。它不是替代推理，而是在推理发生之前，先清理现实。

## 16. 案例二：一个多轮 agent 为什么越聊越笨，以及如何用 state / compaction / memory 修回来

另一个常见场景是：一个研究 agent 在前 3 轮表现很好，到了第 12 轮却开始重复检索、忘记已经确认过的范围、把旧约束和新任务混在一起。使用者往往会说“它记忆衰减了”。这个说法不算完全错，但真正退化的常常不是模型本身，而是对话已经失去结构。

先看最典型的病灶。很多系统把三种完全不同的东西混在一起滚动追加：

- 稳定规则，例如输出规范、工作边界、风险约束；
- 当前状态，例如做到哪一步、什么已经确认、什么仍待决；
- 长期记忆，例如用户偏好、项目背景、常用术语。

一旦三者不分层，每多一轮，就多一层语义缠绕。到后来，模型并不是“忘了”，而是在一个越来越脏的工作记忆里继续推理。

修法通常不是“把历史全部保留”，而是把状态从聊天记录里剥出来。OpenAI 的 conversation state 指南提醒你，多轮可以通过 `previous_response_id` 等机制续接；但这并不意味着上下文管理问题自动消失，因为上下文窗口、输入计费和状态污染仍然存在。真正重要的是：你要有一个独立、短小、可人工检查的状态对象。

例如，一个长期研究 agent 的状态卡可以只有这几项：

```text
目标：完成 AI knowledge base 的 3 个节点扩写
完成：资料已核读；章节结构已定；一个案例已落稿
待办：补诊断清单；补 2 个练习；核来源
约束：只改 3 个文件；不碰未分配文件
下一步：写入并人工复核段落衔接
```

这张卡的作用很像操作系统里的进程控制块。它不负责保存全部记忆，而负责保证当前进程能续接。会话越长，这种状态外显就越重要。

当对话进一步增长时，还要考虑 compaction。OpenAI 的 compaction 指南讲得很直接：长时程交互里，压缩的目标不是单纯省 token，而是在质量、成本和延迟之间重新平衡。这里最值得理解的不是某个 API 参数，而是一条更一般的原则：随着会话变长，你必须主动决定哪些历史应被折叠成状态，哪些证据仍需保留原样，哪些内容应写回外部记忆后移出当前窗口。

很多团队在这一步会犯两个对称错误。第一种错误是什么都不丢，结果每一轮都在带着旧噪音前行；第二种错误是过度摘要，把真正决定判断的引文和边界条件压没了。好的 compaction 不是把历史“变短”，而是把历史“变得仍然可行动”。

最后还要说一句更容易被忽略的话：长期记忆不是聊天记录的垃圾场。只有那些跨任务仍然稳定、未来还会反复用到的事实，才应该进入长期记忆层。把临时偏好、一次性失败、未确认猜测都塞进去，系统得到的不是记忆，而是偏见堆积。

## 17. 诊断清单、人工判断点与练习

到了真正落地的时候，最有用的不是再背几个概念，而是能快速判断“这次系统坏在哪”。下面这个清单可以直接拿来做现场诊断。

| 症状 | 更可能的结构性原因 | 优先修什么 |
| --- | --- | --- |
| 输出泛泛而谈，像在总结“主题”而不是处理“证据” | 证据没有边界，或没有先做引文抽取 | 先重排材料，再拆成两步：抽证据、后判断 |
| 关键信息明明喂进去了，却总被忽略 | 长材料位置与组织方式差，或中段证据埋得太深 | 调整顺序，突出关键片段，必要时做 quote extraction |
| 多轮后反复问过的问题又被重新讨论 | 任务状态没有外显 | 建一个短状态卡，并在每轮显式更新 |
| 费用和延迟越来越高，但质量没有同步变好 | 稳定前缀不稳定，历史无差别累加，缓存/压缩策略缺失 | 稳定前缀、拆动态区、引入 compaction |
| 系统看上去“很会记”，但越来越固执 | 长期记忆层混入临时或错误信息 | 清理长期记忆，只保留跨任务稳定事实 |

这张表之所以重要，是因为它迫使你把故障从“模型不行”翻译成“上下文结构出了什么问题”。这一步翻译本身，就是 context engineering 的核心能力。

### 三个练习

第一，拿一条你过去写过的长 prompt，按 `前缀 / 状态 / 证据 / 记忆 / 回写出口` 五层重新拆一遍。真正的收获不在于改得更漂亮，而在于你会第一次看见自己以前把多少异质信息揉成了一团。

第二，给一个具体任务做“保留原文 / 压成摘要 / 只保留状态”三分法。例如，论文关键段落、用户约束、执行日志、已完成步骤、法规条文、上轮失败原因，各自该进哪一层？只要这一步分不清，系统后面大概率会乱。

第三，故意让模型先在“糟糕上下文”上回答一次，再按本章架构重排后回答第二次。不要只比较语言流畅度，而要比较：引用是否更准确、边界是否更清楚、是否更容易续接下一轮。这种对照实验会让你真正理解：上下文工程优化的不是句子，而是推理条件。

### 最后一个人工判断点

每当你准备把更多材料、更多记忆、更多历史继续塞进窗口前，先停一下问自己：我是在补充证据，还是在转移不确定性？如果答案是后者，那么你大概率不需要更长上下文，而需要更明确的问题定义、更严格的证据选择，或者一次由人来做的裁决。
