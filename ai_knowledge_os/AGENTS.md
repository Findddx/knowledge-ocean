# AI Knowledge OS Agent Rules

## Goal

把新资料编译进一个长期可维护的 AI 知识系统，而不是产出一次性总结。

## Default Output Targets

- 本地 Markdown：完整源文件、模板、协议、长文。
- Notion 数据库：短摘要、标签、状态、回顾队列、跨视图筛选。
- 飞书 Wiki：中文首页、导航层级、长文页面、可阅读入口。

## Source Quality

- 优先使用官方文档、课程主页、论文原文、官方 benchmark 页面、官方比赛页面。
- 二手博客、短视频、搬运帖只能做线索，不能当最终证据。
- 涉及时效性强的能力、产品、价格、竞赛、报告时必须先检索核实。

## Knowledge Layers

- Stable trunk：数学、优化、数据、模型、训练、系统、评估、安全。
- Fast leaves：最新模型、论文、开源项目、比赛、融资、工具更新。
- Applied layer：与具体项目、产品判断、实验路线直接相关的专题。

## Page Types

- Blueprint：顶层结构与设计原则。
- Concept Node：单一概念或方法。
- Workflow：可执行流程。
- Map：某一领域的全景图。
- Resource Index：资料入口与学习源。
- Project Lab：实操项目。
- Radar：前沿变化追踪。
- Market Lens：产品、市场与创业判断。

## Required Template For New Knowledge

每个新资料至少回答：

1. 它属于哪个赛道？
2. 它回应了什么问题压力？
3. 它真正的新意是什么？
4. 哪些内容应进入稳定知识，哪些只进入前沿雷达？
5. 它更新了哪些已有页面？
6. 它带来哪些可复用的项目或判断启发？

## Writing Rules

- 先写问题，再写机制，再写边界，再写行动。
- 解释要能被未来的自己快速重建，不追求华丽措辞。
- 一页只承担一种主要职责，不把地图页写成论文笔记。
- 长文中保留“为什么重要”和“下一步可做什么”。

## Notion Rules

- 数据库卡片只放高信号摘要，不复制整篇长文。
- 属性保持克制，优先保证可筛选和可维护。
- 新建视图只为一种明确任务服务，例如回顾、路线图、项目推进。

## Feishu Rules

- 首页负责解释知识库目标、导航与入口。
- 长文按主题拆页，避免单页过长失控。
- 图片、流程图、白板只在能提高理解效率时使用。

## Anti-Patterns

- 不要把聊天记录原样塞进知识库。
- 不要把论文摘要当作最终知识。
- 不要只堆链接，不写结构。
- 不要先建太复杂的关系型数据库再填内容。
- 不要为了“全面”牺牲可维护性。
