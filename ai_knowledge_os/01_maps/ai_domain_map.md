# AI 顶层知识树

## 1. 地图的用途，不是罗列概念，而是帮助你定向

一张好的领域地图，首先要解决“我现在面对的到底是哪一类问题”。如果它只是把热门名词按栏目排开，那么阅读时也许会有秩序感，真正做研究、做项目、做产品时却仍然会迷路。因为 AI 里的很多问题都长得相似：模型回答得不好，你可以怀疑模型规模、训练数据、后训练、上下文组织、工具接入、检索质量、推理系统、评估集，甚至任务定义本身。地图的意义，就是帮你在这些可能性之间迅速定位。

因此，这张地图不是百科目录，而是一张认知导航图。它试图回答三件事：

1. AI 这个领域的主干结构究竟如何展开。
2. 每个子领域真正关心的中心问题是什么。
3. 当一个系统出问题时，应该先去哪个层面寻找原因。

## 2. 第一层：Foundations 不是“前置知识”，而是约束语言

### 数学与优化

数学在 AI 里最常被误解成“打基础的苦工”。更准确的说法是：数学提供的是约束语言。线性代数告诉你模型如何在向量空间里表示和变换信息；概率统计告诉你不确定性、噪声、似然和泛化如何进入分析；最优化告诉你为什么梯度能带来有效学习，以及为什么有时它又会陷入坏局部、震荡或不稳定。

如果没有这一层，你会把很多经验技巧当作咒语。比如学习率 warmup、归一化、权重衰减、梯度裁剪，看上去是零散工程技巧，实际上都在回应训练动力学中的具体不稳定性。数学层的真正作用，是把“经验感”还原成“结构感”。

### 编程与计算系统

今天的大模型学习，已经不允许把“算法”和“系统”拆开。`CS336` 之所以把 PyTorch、FLOPs、内存、算术强度、GPU、kernel、并行、数据处理与 alignment 放在一条连续课程线上，恰恰因为现代 AI 的能力边界同时由统计规律和计算预算决定。你不知道显存层级，就难以真正理解为什么长上下文昂贵；你不了解并行策略，就很难判断某个模型方案为什么训练得起却部署不起。

这意味着，Foundations 的真正含义不是“先学这些以后再说”，而是：这些概念会在后面每一层反复出现，直到你把它们当成思考母语。

## 3. 第二层：ML / DL Core 研究的是“如何学到表征”

### 机器学习核心

机器学习层讨论的，不只是若干算法名词，而是一个更根本的问题：在有限数据和有限计算下，系统如何从样本中抽取可泛化的结构。监督学习解决的是“给定目标，如何近似”；无监督和自监督学习解决的是“没有显式标签时，世界中有哪些稳定结构可以被利用”；强化学习则把问题变成“当反馈延迟出现时，怎样通过试错获得高价值行为”。

### 深度学习核心

深度学习之所以成为现代 AI 的中心，不在于“层数更多”，而在于它提供了一种可组合的表示学习框架：表示、非线性、归一化、残差连接、优化器、注意力机制，最终共同构成一种可扩展的、可微分的函数族。`Attention Is All You Need` 的重要性，并不仅仅在于提出了 Transformer，更在于它把序列建模中的信息路由问题改写成一种可以大规模并行计算的注意力结构。

### 为什么这层值得单独看

很多学习者会急于跳到 LLM、Agent 或多模态，仿佛“经典深度学习”已经成为历史。这是错觉。今天大模型的所有新现象，仍然在深度学习核心中留下痕迹：注意力如何路由信息，优化器如何塑造收敛轨迹，归一化如何稳定训练，scaling laws 如何描述性能与资源的关系。离开这一层，你会看到很多现象，却无法解释为什么它们会发生。

## 4. 第三层：Data and Training 决定能力如何被灌进系统

### 数据工程

数据不是模型的燃料那么简单，它更像模型世界观的边界条件。采集、清洗、去重、过滤、tokenization、数据配比、数据治理，每一步都在悄悄决定模型最终会对什么敏感、对什么迟钝、对什么偏执。`CS336` 把数据处理放在与架构、并行、alignment 同等重要的位置，原因正在这里：如果输入分布和数据配方有系统偏差，后面的训练技巧只能在偏差之上做局部修补。

### 训练体系

训练并不是一条单线流程，而是一组不同信号的组合。预训练主要解决“世界中有哪些广泛的统计规律可以压缩进参数”；SFT 解决“什么样的行为算作好回答”；DPO、RFT、GRPO 等后训练方法进一步解决“面对偏好、评分、推理和策略型任务时，模型该如何稳定地产生更可取的行为”。

这一层最关键的认知转换是：模型能力不只来自规模，也来自训练信号的形状。很多“看起来像能力提升”的变化，其实只是后训练把已有能力重新压成了更可用的行为；反过来，很多“看起来像工程优化”的决策，最后会深刻影响模型是否学到可靠的策略。

### 这一层的常见错觉

最常见的错觉有两个。第一，把数据问题错认成算法问题。模型答不好，也许不是架构太弱，而是数据压根没覆盖这个问题空间。第二，把 workflow 问题错认成训练问题。很多所谓“该 fine-tune 了”的场景，真正缺的是 context engineering、retrieval、tooling 或 eval，而不是更多梯度更新。

## 5. 第四层：LLM and Foundation Models 把前面的结构汇聚起来

LLM 之所以重要，不只是因为它们“大”，而是因为它们把前面三层的很多线索汇成了一条主河道：大规模自监督学习、Transformer 架构、数据配方、训练优化、推理策略、上下文利用、后训练塑形，最终在一个统一模型家族里相遇。

### Transformer 与位置问题

Transformer 把序列建模变成注意力权重与表示更新问题，但它也带来了新的限制：上下文长度、位置编码、缓存、prefill 成本、注意力复杂度、位置偏置。这些不是后来的“工程细节”，而是模型实际能力边界的一部分。

### Scaling laws 与 compute-optimal training

`Scaling Laws for Neural Language Models` 与 `Chinchilla` 的关键贡献，在于迫使学习者放弃一个朴素直觉：不是模型越大越好，而是参数、数据和算力三者存在结构化关系。这个认识非常重要，因为它改变了我们理解“资源限制”的方式。资源不是单一预算，而是多变量平衡问题。

### 长上下文、推理模型与模型家族

到了这一层，模型不再只是“能补全文本的网络”，而逐渐演化为不同用途的家族：长上下文模型、推理模型、多模态模型、工具使用模型、具备不同延迟与成本轮廓的小模型与大模型。理解 LLM 层的关键，不是记住所有型号，而是理解不同模型设计在“能力、成本、可控性、上下文利用方式”上的结构差异。

## 6. 第五层：Multimodal Generation 让模型接入更多感官与生产流程

多模态层表面上看是“文本之外的扩展”，更深一层看，它是在回答一个问题：世界并不只以文本形式存在，那么模型如何表示图像、视频、音频、界面、动作与时间结构？

图像生成关心的是结构、风格与控制；视频生成额外关心时间一致性、镜头语言、角色连续性；音频与音乐生成关心节奏、音色、编排与条件控制；视觉语言模型则把感知和语言压缩进同一个工作空间。理解这一层时，尤其要避免把 demo 当成 workflow。真正重要的不是“一张图能否生成得漂亮”，而是一个创作流程是否可重复、可控制、可修改、可交付。

## 7. 第六层：Inference Systems and GPU 决定能力能否变成现实成本

训练阶段回答“模型能学到什么”，推理系统回答“这种能力能以什么代价被交付”。这不是附属工程，而是很多产品能否成立的分界线。

### 这里真正研究的是什么

不是单个优化技巧，而是资源协调：batching、KV cache、显存占用、吞吐与延迟权衡、量化、speculative decoding、并行策略、通信成本、服务编排。一个优化若不能明确说明它改善的是哪一个约束，就很可能只是局部技巧。

### 为什么这层必须进入主干知识树

因为现代 AI 很多“能力问题”其实是经济问题。长上下文再强，如果 prefill 成本太高，就无法进入高频生产链路；小模型若成本足够低，即使能力稍弱，也可能在业务上更优。把系统层纳入主干，不是为了让每个人都去写 kernel，而是为了让你知道模型选择从来不是纯算法决定。

## 8. 第七层：RAG、Tools、Agents 让模型不再只在参数里思考

这一层的中心变化，是把模型从“闭门做题”变成“在环境中工作”。RAG 解决的是知识外置问题：当事实变化快、语料体量大、答案需要证据时，不应把所有内容压进参数。Tool use 解决的是动作外置问题：搜索、计算、数据库查询、浏览器操作、代码执行，都可以交给专门工具。Agent 进一步解决的是流程外置问题：规划、执行、校验、反思、状态延续和多步目标管理。

但这层也最容易制造幻觉。很多人看到 Agent 失败，会立即把原因归咎为“模型不够聪明”。实际上，失败常常来自更普通的地方：工具接口模糊、检索内容无关、上下文过噪、状态管理差、没有 guardrails、没有 eval、没有人类复核点。RAG、tools、agents 不应被看作“更酷的能力”，而应被看作“让模型接入外部资源和工作流的不同方式”。

## 9. 第八层：Evaluation / Safety / Governance 是整个系统的刹车与仪表盘

如果前面的层次主要在回答“怎样做得出来”，这一层回答的是“怎样知道它真的做对了，而且值得部署”。评估、benchmark、安全、治理，不是最后才补上的装饰，而是所有前面工作的校验接口。

一个没有 eval 的工作流，本质上是在用主观满意度替代证据；一个没有 guardrails 的 agent，本质上是在把流程权力无条件外包给概率模型；一个没有治理视角的产品判断，最终会在合规、信任、使用场景或责任边界上付出更高代价。越往后走，判断越重要，而判断必须建立在可解释的评估与约束之上。

## 10. 第九层：Applications / Product / Market 把技术变成价值判断

AI 知识树的终点，不是“知道更多模型”，而是能够回答真实世界的问题：用户是谁，原工作流是什么，AI 改变了哪一步，价值如何被验证，数据闭环在哪里，护城河落在模型、数据、流程、分发还是组织能力上。

这也是为什么 `The Full Stack` 这类课程一直强调：构建 AI 产品不仅是训练模型或写 prompt，而是贯穿问题定义、模型和 GPU 选择、部署、持续学习与用户体验设计的完整生命周期。技术能力如果不能进入产品与市场判断，知识树就会停在实验室。

## 11. 如何使用这张地图，而不是被它压住

地图越大，越容易让人产生另一种焦虑：什么都重要，于是什么都不敢深入。避免这种焦虑的方式，不是缩小地图，而是学会按问题进入。

### 如果你在问“模型为什么会这样学”

优先回到 `Foundations -> ML / DL Core -> Data and Training`。这里讨论的是表征、优化、数据与训练信号，是能力形成的上游。

### 如果你在问“系统为什么这么贵、这么慢、这么不稳定”

优先回到 `Inference Systems and GPU -> Context / Retrieval / Tools`。这里讨论的是资源、上下文和流程，而不是再去换一个更大的模型。

### 如果你在问“这个方向值不值得做成产品或长期主线”

优先联动 `Evaluation / Safety / Governance -> Applications -> Product / Market`。这里的关键不是技术新奇，而是价值闭环、可控性和成本结构。

## 12. 这张地图最终导向哪三类产物

地图不是为了停留在地图上。每一块知识最终都应当导向三类产物中的至少一种：

1. `Concept Nodes`：把一个机制讲清楚。
2. `Project Labs`：把一个问题做出来。
3. `Market Lenses`：把一个方向判断明白。

只有当地图不断流向这三类产物，它才不是“知识陈列”，而是“知识生产”。

## 13. 读这张地图时，最重要的动作是“逆向追因”

很多人拿到领域地图后的第一反应，是顺着热点往前跑：最近流行什么，就先点开什么。这样当然会带来短期兴奋感，但很难真正形成结构判断。更有力的读法，往往是反过来：当一个系统失败时，从失败点往回追，看看它究竟属于哪一层问题。

这张地图最有用的时候，不是你在空闲时浏览，而是你遇到一个具体故障、具体机会或具体路线选择的时候。

### 小案例一：一个 research agent 接了更多工具，为什么反而更笨

表面上看，问题好像出在 `Agents` 层：它不会规划，或者不会调用工具。可如果你真的沿地图逆向追，会发现至少有四个可能的更深层原因：

- `Context / Retrieval` 层：检索回来的材料并不真正相关，agent 只是很努力地在噪声中工作。
- `Workflow / State` 层：每一轮都在重新解释任务，没有稳定状态，没有压缩后的中间结论。
- `Evaluation` 层：系统根本没有定义什么叫“研究完成”，于是 agent 只能用语言流畅度冒充进展。
- `Foundations of the task`：问题被定义得过于含混，本来就不存在清晰的可执行目标。

这就是地图的价值所在：它逼你承认，越像“智能”的故障，往往越不应只在“智能”那一层找原因。

### 小案例二：GUI agent 的 demo 很惊艳，产品却并不稳

这类情况很容易让人误判成“模型离 AGI 还远”。更准确的说法往往是：`computer use` 任务把好几层难题叠在了一起。

- `Multimodal` 层：模型要看懂界面、文本、视觉提示与空间关系。
- `Inference / Systems` 层：延迟和成本一上来，交互体验立刻恶化。
- `Agents` 层：多步操作需要状态延续、回退机制和错误恢复。
- `Evaluation` 层：demo 的成功标准和真实组织工作流的成功标准完全不是一回事。

OSWorld 这类 benchmark 的意义，就在于把“看起来会操作电脑”拆成更细的能力结构，而不是让我们停留在单次演示的震撼里。

### 小案例三：更大的模型分数更高，为什么产品上反而该选更小的模型

这不是反常，而是地图在提醒你：`Model capability` 和 `Product choice` 从来不是同一个层级的问题。

更大的模型也许在 benchmark 上更强，但一旦进入产品：

- `Inference Systems` 会问：它的延迟、成本、吞吐是否可接受？
- `Workflow` 会问：这一步真的需要最强模型，还是可以只在关键节点升级？
- `Product / Market` 会问：用户是为那一点能力增量付费，还是为更快、更稳、更便宜付费？

当你顺着地图看问题，就会知道：很多“技术上更强”的方案，在系统和商业上未必更优。

## 14. 四类学习者，怎样进入这张地图

### 研究型学习者：先把上游讲清楚

如果你的目标是研究、理论理解或长期技术判断，那么最应该反复走的是：

`Foundations -> ML / DL Core -> Data and Training -> Foundation Models`

这条路的价值，不在于它更“纯”，而在于它最接近能力形成的上游。你会更容易看清哪些现象来自表示学习、哪些来自训练信号、哪些只是后训练把已有能力重新整形。

### 构建型学习者：中游和下游要一起看

如果你的目标是做 agent、做工作流、做应用，那么更常用的路径会是：

`Foundation Models -> Inference Systems -> Retrieval / Tools / Agents -> Evaluation`

这条路径的关键，不是“赶快学会搭东西”，而是避免把所有问题都交给模型本身。真正稳定的构建者，往往最善于把问题拆到系统、工具和评估层。

### 系统型学习者：别把性能工程看成外围

如果你对 serving、GPU、吞吐、延迟、KV cache、并行策略更敏感，那么最重要的路径是：

`Foundations -> Foundation Models -> Inference Systems -> Product economics`

你最终会发现，系统工作并不是技术主干旁边的后勤，而是在决定“什么能力能以什么代价被现实使用”。

### 产品与创业型学习者：从价值闭环往回读

如果你的问题是“这个方向值不值得押”“什么东西可能变成长期能力”，那么入口往往是：

`Applications / Product / Market -> Evaluation / Safety / Governance -> Systems -> Models`

这不是说可以跳过技术，而是说你的核心问题首先是价值判断。只有当价值、可控性和成本结构成立时，技术优势才真正有落点。

## 15. 讨论题与用法练习

1. 找一个你最近见过的 AI 产品或 demo，用这张地图判断它的主要难点究竟位于哪三层，而不是直接说“模型还不够强”。
2. 回想一次你自己的系统失败，倒着沿地图追溯：它最初看上去像哪一层问题，最后真正根因在哪一层？
3. 如果你要在未来 3 个月深挖一个方向，请说明你会从地图上的哪一层进入，以及你为什么不从别的层进入。
